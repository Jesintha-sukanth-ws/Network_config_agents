================================================================================
                        NETWORK ORCHESTRATION SYSTEM
                           ARCHITECTURE DOCUMENTATION
================================================================================

OVERVIEW:
This system provides intelligent network orchestration for switch configuration
tasks. It processes natural language task descriptions and converts them into
validated, executable network configurations through a 6-step pipeline.

================================================================================
                              COMPLETE FLOW DIAGRAM
================================================================================

[USER INPUT] → [ORCHESTRATOR] → [6-STEP PIPELINE] → [TERMINAL OUTPUT]

STEP 1: Intent Extraction
    Input:  Natural language task description
    Process: LLM (Ollama) converts text to structured JSON workflow
    Output: Validated intent workflow with steps and parameters

STEP 2: Schema Validation  
    Input:  Intent workflow JSON
    Process: Validates structure, intent types, parameters, datatypes
    Output: Normalized workflow with validated parameters

STEP 3: Workflow Validation
    Input:  Normalized workflow
    Process: VLAN ranges, interface formats, trunk configurations
    Output: Workflow validation result (safe/errors)

STEP 4: CMDB Lookup
    Input:  Configuration Item (CI) sys_id from task
    Process: ServiceNow API call to retrieve device metadata
    Output: Device details (name, vendor, model, management host, OS type)

STEP 5: Device State Retrieval
    Input:  Device metadata from CMDB
    Process: API calls (NX-API/RESTCONF) to get current device state
    Output: Current VLANs, interfaces, trunk configurations

STEP 6: State Validation
    Input:  Desired workflow + current device state
    Process: Idempotency checking, dependency validation
    Output: Execution plan with skip recommendations

FINAL: Terminal Display
    Input:  Complete orchestration result
    Process: Format and display comprehensive results
    Output: Formatted terminal output with all details

================================================================================
                              CORE SERVICES LAYER
================================================================================

📄 app/services/orchestrator_service.py
ROLE: Main workflow coordinator and pipeline orchestrator
INPUT: Task data dictionary with description and CI information
PROCESS:
  - Coordinates all 6 pipeline steps
  - Manages error handling and flow control
  - Integrates all validation layers
  - Provides detailed progress logging
OUTPUT: Complete orchestration result with execution plan
DEPENDENCIES: All other services and validators
KEY FUNCTIONS:
  - process_task(task_data) → Main entry point

📄 app/services/intent_service.py  
ROLE: Natural language to structured intent conversion
INPUT: Natural language task description (string)
PROCESS:
  - Sends description to Ollama LLM with system prompt
  - Parses JSON response into structured workflow
  - Validates against Pydantic models
OUTPUT: Validated intent workflow JSON or error
DEPENDENCIES: Ollama, intent_models, prompts
KEY FUNCTIONS:
  - parse_intent(task_description) → Intent workflow JSON

📄 app/services/cmdb_service.py
ROLE: Device metadata retrieval from ServiceNow CMDB
INPUT: Configuration Item (CI) sys_id
PROCESS:
  - Makes REST API call to ServiceNow CMDB
  - Retrieves device metadata (name, vendor, model, OS, management host)
  - Resolves manufacturer sys_id to vendor name
OUTPUT: Device metadata dictionary or error
DEPENDENCIES: ServiceNow API, config/settings
KEY FUNCTIONS:
  - get_cmdb_data(ci_sys_id) → Device metadata

📄 app/services/display_service.py
ROLE: Terminal output formatting and presentation
INPUT: Complete orchestration result dictionary
PROCESS:
  - Formats task information, device details, workflow steps
  - Displays device state summary and relevant interfaces
  - Shows execution status and warnings
OUTPUT: Formatted terminal output (printed to console)
DEPENDENCIES: None (pure formatting)
KEY FUNCTIONS:
  - display_terminal_output(result) → Formatted console output

================================================================================
                            VALIDATION LAYERS
================================================================================

📄 app/validation/schema_validator.py
ROLE: Workflow structure and parameter validation
INPUT: Intent workflow JSON from LLM
PROCESS:
  - Validates workflow structure (must be dict with workflow list)
  - Checks intent types against registry (14 supported intents)
  - Validates required/optional parameters
  - Checks parameter datatypes (int, str, list)
  - Resolves intent aliases
OUTPUT: Validation result (safe/errors) + normalized workflow
DEPENDENCIES: switch_intent_schema_registry
KEY FUNCTIONS:
  - validate_workflow(workflow_json) → Validation result
  - resolve_alias(intent_type) → Canonical intent name

📄 app/network_validation/workflow_validator.py
ROLE: Network-specific workflow validation orchestrator
INPUT: Normalized workflow from schema validator
PROCESS:
  - Routes VLAN intents to VlanValidator
  - Routes interface intents to InterfaceValidator  
  - Routes trunk intents to TrunkValidator
  - Aggregates all validation errors
OUTPUT: Unified validation result (safe/errors)
DEPENDENCIES: VlanValidator, InterfaceValidator, TrunkValidator
KEY FUNCTIONS:
  - validate_workflow(workflow) → Validation result

📄 app/network_validation/state_validator.py
ROLE: Device state comparison and idempotency checking
INPUT: Desired workflow + current device state
PROCESS:
  - Compares desired state vs current state for each step
  - Detects already-configured items (idempotency)
  - Validates VLAN dependencies exist
  - Builds execution plan with skip recommendations
OUTPUT: State validation result + execution plan
DEPENDENCIES: base_validator
KEY FUNCTIONS:
  - validate_state(workflow, current_state) → Validation + execution plan

📄 app/network_validation/vlan_validator.py
ROLE: VLAN-specific validation rules
INPUT: VLAN intent parameters
PROCESS:
  - Validates VLAN ID ranges (1-4094, excluding reserved)
  - Checks VLAN name format and length
  - Validates VLAN operations (create, delete, rename)
OUTPUT: VLAN validation errors list
KEY FUNCTIONS:
  - validate_vlan_operation(intent_type, parameters, step_number) → Errors

📄 app/network_validation/interface_validator.py
ROLE: Interface-specific validation rules  
INPUT: Interface intent parameters
PROCESS:
  - Validates interface name formats (Gi1/0/1, Eth1/1, etc.)
  - Checks interface operations (mode, description, shutdown)
  - Validates interface parameter combinations
OUTPUT: Interface validation errors list
KEY FUNCTIONS:
  - validate_interface_operation(intent_type, parameters, step_number) → Errors

📄 app/network_validation/trunk_validator.py
ROLE: Trunk-specific validation rules
INPUT: Trunk intent parameters  
PROCESS:
  - Validates trunk VLAN lists and ranges
  - Checks native VLAN configurations
  - Validates allowed VLAN lists
OUTPUT: Trunk validation errors list
KEY FUNCTIONS:
  - validate_trunk_operation(intent_type, parameters, step_number) → Errors

📄 app/network_validation/base_validator.py
ROLE: Common validation utilities and error formatting
INPUT: Various validation parameters
PROCESS:
  - Provides standard error formatting
  - Common validation helper functions
  - Base class for all validators
OUTPUT: Standardized error objects
KEY FUNCTIONS:
  - build_error() → Standardized error format
  - build_result() → Standardized result format

================================================================================
                              DATA FLOW DETAILS
================================================================================

🔄 STEP 1: INTENT EXTRACTION
Input Format:
  "Configure VLAN 100 named FINANCE on interface Gi1/0/5"

Processing:
  1. orchestrator_service.py calls parse_intent()
  2. intent_service.py sends to Ollama LLM with system prompt
  3. LLM returns structured JSON workflow
  4. Pydantic models validate the structure

Output Format:
  {
    "workflow": [
      {
        "intent_type": "create_vlan",
        "parameters": {"vlan_id": 100, "name": "FINANCE"}
      },
      {
        "intent_type": "assign_access_vlan", 
        "parameters": {"interface": "Gi1/0/5", "vlan_id": 100}
      }
    ]
  }

🔄 STEP 2: SCHEMA VALIDATION
Input: Intent workflow JSON from Step 1
Processing:
  1. schema_validator.py validates workflow structure
  2. Checks each step against switch_intent_schema_registry.py
  3. Validates required parameters exist
  4. Checks parameter datatypes (int, str, list)
  5. Resolves any intent aliases to canonical names

Output Format:
  {
    "safe": true,
    "errors": [],
    "workflow": [normalized_workflow_steps]
  }

🔄 STEP 3: WORKFLOW VALIDATION  
Input: Normalized workflow from Step 2
Processing:
  1. workflow_validator.py routes intents to specific validators
  2. vlan_validator.py checks VLAN ranges (1-4094)
  3. interface_validator.py validates interface formats
  4. trunk_validator.py checks trunk configurations
  5. All errors aggregated into unified result

Output Format:
  {
    "safe": true/false,
    "errors": [list_of_validation_errors]
  }

🔄 STEP 4: CMDB LOOKUP
Input: CI sys_id from task data
Processing:
  1. cmdb_service.py makes REST API call to ServiceNow
  2. Retrieves device metadata fields
  3. Resolves manufacturer sys_id to vendor name
  4. Returns structured device information

Output Format:
  {
    "device_name": "SW-CORE-01",
    "vendor": "Cisco", 
    "model": "C9300-48P",
    "os_type": "IOS-XE",
    "os_version": "16.12.05",
    "management_host": "192.168.1.100"
  }

🔄 STEP 5: DEVICE STATE RETRIEVAL
Input: Device metadata from Step 4
Processing:
  1. device_state_service.py determines vendor/OS type
  2. connection_service.py builds connection parameters
  3. Makes API calls (NX-API for NX-OS, RESTCONF for IOS-XE)
  4. Normalizes vendor responses to common format
  5. Returns current VLANs, interfaces, trunks

Output Format:
  {
    "device_info": {"os_version": "...", "hostname": "..."},
    "vlans": [{"vlan_id": 10, "name": "MGMT", "state": "active"}],
    "interfaces": [{"name": "Gi1/0/1", "mode": "access", "access_vlan": 10}],
    "trunks": [{"interface": "Gi1/0/48", "allowed_vlans": [10,20]}]
  }

🔄 STEP 6: STATE VALIDATION
Input: Desired workflow + current device state
Processing:
  1. state_validator.py compares desired vs current state
  2. For each workflow step, checks if already configured
  3. Validates VLAN dependencies exist
  4. Builds execution plan with skip recommendations
  5. Returns idempotency analysis

Output Format:
  {
    "safe": true,
    "errors": [],
    "execution_plan": [
      {
        "step": 1,
        "intent_type": "create_vlan",
        "execute": false,
        "reason": "VLAN 100 already exists"
      }
    ]
  }

🔄 FINAL: TERMINAL DISPLAY
Input: Complete orchestration result
Processing:
  1. display_service.py formats all collected data
  2. Shows task info, device details, workflow steps
  3. Displays device state summary and execution plan
  4. Prints comprehensive formatted output

Output: Formatted terminal display with all orchestration details

================================================================================
                              ERROR HANDLING
================================================================================

🚨 ERROR PROPAGATION FLOW:
1. Any step failure stops the pipeline
2. Errors are captured with detailed context
3. Error details include step, intent type, parameter info
4. User receives specific error messages for debugging

🚨 COMMON ERROR TYPES:
- intent_extraction_failed: LLM parsing errors
- schema_validation_failed: Invalid workflow structure
- workflow_validation_failed: Network validation errors  
- cmdb_lookup_failed: ServiceNow API errors
- device_facts_failed: Device connectivity errors
- state_validation_failed: State comparison errors

🚨 ERROR RECOVERY:
- Validation errors provide specific fix guidance
- Device connectivity errors suggest credential checks
- CMDB errors indicate CI or network issues
